Queues, and the skills on each transition

Columns are resting states — the name is true of every card in them, always. Agents run on the transitions: a drag is the launch. Code is the live activity lane holding the whole build-and-review cycle — the review chains automatically after the build, and a card can only earn its way out. Deploy is not a column at all: it is a terminal drop zone — dropping a reviewed card merges the work into main once CI passes and closes the ticket, so shipped work simply leaves the board. State on the column, verb on the drop target.

IDEAS ➜ PRODUCT BACKLOG
Drag opens the ideation chat (evolve mode, guided): an in-daemon agent challenges the idea, rewrites the same ticket, and removes the human/idea label. no marker — the label swap moves the card
PRODUCT ➜ ENGINEERING BACKLOG
Launches /human-plan <KEY>: planner + verify agents draft the engineering plan; it lands as a [human:plan] comment. While it runs the card stays here with ◐ planning…. [human:planning-started] → [human:plan-ready]
ENGINEERING ➜ CODE — BUILD, THEN AUTO-REVIEW
Launches /human-execute <KEY>: the executor works the plan and runs the human-done self-check. When the handoff posts, the daemon chains /human-review automatically — the whole build-and-review cycle lives in Code. [human:implementation-started] → [human:ready-for-review] → [human:review-started]
CODE ➜ READY TO DEPLOY — EARNED, NOT DRAGGED
A passing verdict releases the card by itself. A failing verdict pins it in Code with ⚠ — the findings land as a ticket comment, and re-dropping the card onto Code rebuilds against them. [human:review-complete] verdict: pass | fail
DEPLOY — A DROP ZONE, NOT A COLUMN
Dropping a reviewed card ships it: push → PR → CI green → merge → branch deleted → ticket closed. The card leaves the board. On merge-deploy platforms (Scalingo, Heroku, Vercel) this IS the production deploy. [human:deploy-started] → [human:deployed]
